# 12 — Observability: Double-Entry Ledger Service

---

## Objective

Define the logging, tracing, metrics, alerting, and SLO strategy for the double-entry ledger. Financial services demand high-confidence observability — every posting must be traceable from API call to journal entry to Kafka event.

---

## Observability Pillars

| Pillar | Tool | Purpose |
|---|---|---|
| Structured Logging | Logback + ELK (Elasticsearch, Logstash, Kibana) | Audit trail, debugging, correlation |
| Distributed Tracing | OpenTelemetry + Jaeger / AWS X-Ray | Request flow across services |
| Metrics | Micrometer + Prometheus + Grafana | Performance, health, SLO tracking |
| Alerting | Alertmanager + PagerDuty | On-call notification for SLO breach |
| Financial Integrity Checks | Custom Prometheus gauges | Invariant monitoring |

## Structured Logging Strategy

### Log Levels

| Level | When Used |
|---|---|
| ERROR | Debit=credit invariant violation, DB commit failure, security violation |
| WARN | CAS retry on snapshot, idempotency cache miss, slow DB query (> 100ms) |
| INFO | Posting created (with posting_id, account_ids, amounts), reversal posted |
| DEBUG | Account validation steps, snapshot delta computation — disabled in production |

### Log Format

Every log line is JSON (structured) and includes:

```json
{
  "timestamp": "2024-01-15T10:30:00.123Z",
  "level": "INFO",
  "service": "ledger-service",
  "pod": "ledger-7d4f8b9c6-xk2p9",
  "trace_id": "4bf92f3577b34da6",
  "span_id": "00f067aa0ba902b7",
  "request_id": "req_uuid",
  "posting_id": "post_uuid",
  "idempotency_key": "payment:pay_xyz789",
  "caller_service": "payment-service",
  "event": "POSTING_CREATED",
  "account_ids": ["acc_src", "acc_dst"],
  "total_debit": 500000,
  "currency": "INR",
  "duration_ms": 18
}
```

**Sensitive field masking:** `account_id` values are kept (necessary for audit trail). User PII in metadata is masked: `"initiated_by": "usr_***abc"` (show last 3 chars only).

### Correlation IDs

- `trace_id`: Propagated from caller via W3C `traceparent` header (OpenTelemetry standard)
- `request_id`: Generated at the API gateway, flows through the entire request
- `posting_id`: Generated by the ledger service, referenced in all downstream Kafka events

A finance ops team can trace: "this payment → this posting → these journal entries → these Kafka events → fraud assessment" using the `posting_id` as the join key across all systems.

## Distributed Tracing

**Instrument:**
- Spring Boot auto-instrumented via `spring-boot-starter-actuator` + `micrometer-tracing-bridge-otel`
- All DB queries traced via JDBC instrumentation (OpenTelemetry JDBC plugin)
- Kafka produce/consume operations traced via Kafka instrumentation

**Trace structure for a posting:**

```
POST /postings (root span: 18ms)
├── IdempotencyGuard.check (2ms)
│   ├── Redis GET (1ms)
│   └── DB SELECT (0ms — Redis hit)
├── PostingService.validateLegs (1ms)
├── AccountRepository.findAllById (5ms)
│   └── DB SELECT (4ms)
├── DB Transaction (9ms)
│   ├── INSERT postings (2ms)
│   ├── INSERT journal_entries batch (4ms)
│   ├── UPDATE account_snapshots x2 (2ms)
│   └── INSERT outbox_events (1ms)
└── Redis cache invalidation (1ms — async)
```

**Sampling rate:**
- Production: 1% trace sampling (sufficient for P99 analysis at 5,000 RPS = 50 full traces/sec)
- On error: 100% sampling (all error requests traced)
- On slow request (> 100ms): 100% sampling (head-based + tail-based sampling with OpenTelemetry Collector)

## Metrics (Prometheus + Micrometer)

### Core Business Metrics

```
# Postings
ledger_postings_total{status="success|failure", reference_type="PAYMENT|LOAN|..."} counter
ledger_postings_duration_seconds{quantile="0.5|0.95|0.99"} histogram
ledger_postings_legs_total histogram (distribution of legs per posting)

# Idempotency
ledger_idempotency_hits_total counter (duplicate requests caught)
ledger_idempotency_misses_total counter (new requests)

# Balance reads
ledger_balance_reads_total{source="cache|snapshot|computed"} counter
ledger_balance_read_duration_seconds histogram

# Snapshot
ledger_snapshot_cas_retries_total counter
ledger_snapshot_staleness_seconds gauge (age of oldest snapshot)

# Outbox
ledger_outbox_unpublished_events gauge
ledger_outbox_relay_duration_seconds histogram
ledger_outbox_publish_failures_total counter

# Financial integrity
ledger_invariant_violations_total counter (debit != credit — should always be 0)
ledger_snapshot_mismatch_accounts gauge (accounts where snapshot != journal sum)
```

### Infrastructure Metrics

```
# DB
db_connections_active gauge
db_connections_pending gauge
db_query_duration_seconds{query_type="posting_insert|balance_select|..."} histogram
db_replica_lag_seconds gauge

# Redis
redis_operations_total counter
redis_miss_rate gauge
redis_memory_usage_bytes gauge
redis_evicted_keys_total counter

# Kafka
kafka_outbox_lag gauge (outbox_events published=false count)
kafka_consumer_lag{consumer_group, topic, partition} gauge
kafka_producer_error_rate gauge
```

## Dashboards (Grafana)

### Dashboard 1: Posting Health

| Panel | Metric | Alert |
|---|---|---|
| Posting throughput | `rate(ledger_postings_total[1m])` | < 100 RPS when expected > 1000 |
| P99 posting latency | `histogram_quantile(0.99, ledger_postings_duration_seconds)` | > 200ms |
| Failure rate | `rate(ledger_postings_total{status="failure"}[1m])` | > 1% |
| Idempotency hit rate | `ledger_idempotency_hits / ledger_postings_total` | Sudden spike (replay attack) |

## Alerting Strategy

### Critical Alerts (P0 — Page on-call immediately)

| Alert | Condition | Action |
|---|---|---|
| Invariant violation | `ledger_invariant_violations_total > 0` | Immediate incident — check for corrupt posting |
| Snapshot mismatch detected | `ledger_snapshot_mismatch_accounts > 0` | Switch balance reads to journal computation |
| DB primary unreachable | `db_connections_active == 0 for 10s` | Initiate DB failover |
| Posting failure rate | `> 5% for 1 minute` | Investigate DB, Redis, idempotency |

### High Alerts (P1 — Page on-call within 15 minutes)

| Alert | Condition |
|---|---|
| Posting P99 > 500ms | Sustained for 5 minutes |
| Outbox lag > 50,000 events | Growing, not shrinking |
| Snapshot CAS retry rate > 20% | Hot account contention |
| DB replica lag > 1 second | Risk of stale balance reads |
| Redis memory > 85% | Risk of eviction affecting idempotency |

### Warning Alerts (P2 — Slack notification)

| Alert | Condition |
|---|---|
| Balance cache hit rate < 90% | Cache efficiency degraded |
| Kafka consumer lag growing | Consumer group falling behind |
| Idempotency hit spike | Possible retry storm from upstream |

## SLI / SLO Definitions

| SLI | Target SLO | Measurement Window |
|---|---|---|
| Posting success rate | 99.9% | Rolling 30 days |
| Posting P99 latency < 200ms | 99% of posting requests | Rolling 7 days |
| Balance read success rate | 99.99% | Rolling 30 days |
| Balance read P99 latency < 100ms | 99.5% of balance reads | Rolling 7 days |
| Outbox relay lag < 5 seconds | 99.5% of events | Rolling 24 hours |
| Invariant integrity | 100% — zero violations ever | Continuous |

**Error budget:** At 99.9% posting success, the 30-day error budget = 43.2 minutes of tolerable downtime. Alert when 50% of error budget consumed in < 5 days.

## Runbooks

### Runbook: High Snapshot CAS Retry Rate

1. Check `ledger_snapshot_cas_retries_total` — which account_id is retrying most?
2. Query: `SELECT account_id, COUNT(*) FROM posting_snapshot_retries GROUP BY account_id ORDER BY count DESC LIMIT 10`
3. If a single account is dominant: it's a hot account — escalate to engineering for virtual account sharding
4. Temporary mitigation: disable in-transaction snapshot update for the hot account; snapshot updated asynchronously every 30 seconds

### Runbook: Snapshot Mismatch Alert

1. Alert fires: `ledger_snapshot_mismatch_accounts > 0`
2. Identify affected accounts from the mismatch report
3. Switch balance reads for affected accounts from snapshot to `SUM(journal_entries)` (slower, correct)
4. Trigger snapshot rebuild job for affected accounts
5. Verify rebuild: re-run mismatch check
6. Root cause analysis: review recent deployments, CAS logic changes

## Interview Discussion Points

- **How do you audit a specific payment end-to-end?** Search by `idempotency_key` (e.g., `payment:pay_xyz789`) in Kibana: find posting_id. Query `/api/v1/postings/{posting_id}` for all legs. Search Jaeger traces by `posting_id`. Search Kafka `ledger.audit.events` topic by `posting_id`. Complete lineage from API call to journal entries to downstream events
- **How do you detect a financial integrity issue proactively?** Scheduled job runs every 15 minutes: `SELECT account_id FROM accounts WHERE ABS((SELECT SUM(CASE direction WHEN 'DEBIT' THEN amount ELSE -amount END) FROM journal_entries WHERE account_id = accounts.account_id) - (SELECT balance FROM account_snapshots WHERE account_id = accounts.account_id)) > 0`. Any non-zero result triggers a P1 alert
- **What metrics would a CISO care about?** Authorization failure rate (spike = brute force), idempotency key collision pattern (replay attack detection), admin login frequency (privilege escalation monitoring), detokenization attempts per service per minute
- **How do you calculate the error budget?** SLO = 99.9% for postings. In 30 days: 30 × 24 × 60 = 43,200 minutes. 0.1% × 43,200 = 43.2 minutes tolerable error. Track `(total_postings - failed_postings) / total_postings` in a 30-day rolling window. Burn rate alert: if error rate is 5x the SLO (0.5%) for 1 hour, the monthly error budget will be exhausted in 5 days → page
