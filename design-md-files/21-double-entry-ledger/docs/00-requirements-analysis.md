# 00 — Requirements Analysis: Double-Entry Ledger Service

---

## Objective

Define functional and non-functional requirements, constraints, assumptions, and capacity estimates for a production-grade double-entry ledger service — the financial core of any banking, fintech, or payment platform.

---

## Functional Requirements

### Core (MVP)

- Every transaction posts as two or more journal entries: total debits must equal total credits (the fundamental invariant)
- Support atomic multi-leg postings (e.g., split a transaction across three accounts in one atomic operation)
- Accounts have a type: ASSET, LIABILITY, EQUITY, INCOME, EXPENSE (standard chart of accounts)
- Compute current balance for any account on demand
- Assign a globally unique idempotency key per posting to prevent duplicate entries
- Support debit and credit directions relative to account type (e.g., a debit increases an ASSET account but decreases a LIABILITY account)

### Extended (V1)

- Point-in-time balance queries: what was this account's balance at timestamp T?
- Multi-currency: store amounts with currency code; FX conversion entries recorded as separate legs
- Balance snapshots: periodic materialized balance to avoid recomputing from journal on every read
- Audit trail: every entry is immutable; deletions are compensating entries (reversals)
- Reconciliation API: compare ledger balance against external source for a given account and date range

### Advanced (V2+)

- Ledger sharding by account range or tenant (for multi-tenant fintech platforms)
- Streaming balance updates via Kafka for downstream consumers (analytics, risk, fraud)
- GL (General Ledger) report generation: trial balance, balance sheet, P&L
- Soft-close periods: prevent backdated entries beyond a closed accounting period
- Multi-entity support: holding company with subsidiary ledgers that roll up

## Non-Functional Requirements

| Property | Target |
|---|---|
| Consistency | Strong — debit=credit invariant must never be violated |
| Availability | 99.99% — ledger is the financial source of truth |
| Posting latency (P99) | < 200ms for single-currency single-leg pair |
| Balance query latency (P99) | < 100ms with snapshot strategy |
| Durability | Zero tolerance — no posted entry must be lost |
| Audit | Every entry immutable and traceable to originating event |
| Throughput | 5,000 postings/sec sustained; 20,000 burst |
| Data retention | 7+ years (regulatory requirement in most jurisdictions) |

---

## Assumptions

- The ledger is an internal service — upstream services (payment, wallet, loan) call it directly; end users never interact with it
- The system is authoritative: it is the single source of truth for financial balances
- FX rates are injected by the caller; this service does not fetch exchange rates
- Idempotency keys are generated by the caller and passed in the request
- The chart of accounts is managed externally (account creation is a separate concern in V1)
- All monetary amounts are stored as integers (cents, paise, or satoshis) — no floating point arithmetic
- Regulatory period-close enforcement is out of scope for MVP but flagged for V2

---

## Constraints

- No floating-point arithmetic anywhere in the money path — use `Long` (BigDecimal allowed for display only)
- Every journal entry is append-only — no UPDATE or DELETE on journal entries table
- Balances derived from journal entries must be provably consistent with the journal (verifiable by replay)
- All operations within a posting must commit atomically — partial postings are a financial error state
- The service must be idempotent: replaying the same idempotency key returns the existing result, not a duplicate

## Scale Estimation

### Traffic Assumptions

| Metric | Value |
|---|---|
| Postings per day | 20 million |
| Avg legs per posting | 2.5 (many 2-leg, some 3-leg splits) |
| Journal entries per day | 50 million |
| Balance queries per day | 200 million (read-heavy downstream) |
| Peak posting RPS | 5,000 |
| Peak balance read RPS | 50,000 |
| Read:Write ratio | 40:1 (heavily read for balance checks) |

### Back-of-the-Envelope Calculations

**Journal Entry Row Size**

| Field | Size |
|---|---|
| entry_id (UUID) | 16 bytes |
| posting_id (UUID) | 16 bytes |
| account_id (UUID) | 16 bytes |
| direction (DEBIT/CREDIT enum) | 1 byte |
| amount (bigint cents) | 8 bytes |
| currency (char 3) | 3 bytes |
| effective_at (timestamptz) | 8 bytes |
| created_at (timestamptz) | 8 bytes |
| metadata (jsonb) | ~100 bytes |
| **Total** | **~176 bytes** |

- 50M entries/day × 176 bytes ≈ **8.8 GB/day raw**
- 7 years: ~22 TB — partitioning and tiered storage mandatory
- Index overhead: ~50% on top → ~33 TB total with indexes

**Balance Snapshot Table**

- 10 million accounts × 200 bytes/snapshot ≈ 2 GB — small enough to keep hot
- Snapshot refresh: incremental delta on every posting — not full recompute

**Cache (Redis)**

- Hot accounts (top 10%): 1M accounts × 200 bytes ≈ **200 MB** — trivial Redis footprint
- Balance cache TTL: short (1–5 seconds) or invalidation-based on posting events

## Read/Write Patterns

| Operation | Pattern | Frequency |
|---|---|---|
| Post journal entries | Atomic multi-row INSERT | High |
| Query current balance | Snapshot read or aggregated SUM | Very High |
| Point-in-time balance | Filtered SUM up to timestamp | Medium |
| Idempotency check | Lookup by idempotency_key | Every posting |
| Reconciliation | Range scan on account + date | Low, batch |
| GL report generation | Large aggregation across all accounts | Low, scheduled |

**Key Insight**: Balance query is the dominant read path. Must be served from a snapshot or cache — computing `SUM(debit) - SUM(credit)` across millions of entries on every request is unacceptable at scale.

## Tradeoffs Acknowledged at Requirements Level

| Decision | Tradeoff |
|---|---|
| Strong consistency over availability | CAP: choose CP — a ledger split-brain creates financial errors |
| Append-only journal | No UPDATE/DELETE = large storage growth but complete audit trail |
| Integer arithmetic only | Slightly complex display logic, but eliminates rounding errors |
| Snapshot for balance | Stale risk (mitigated by invalidation), but eliminates expensive aggregation |
| Idempotency via key lookup | Extra SELECT on every POST, but prevents duplicate financial entries |
| Point-in-time queries | Powerful audit capability but requires timestamps on every entry |

## Interview Discussion Points

- **Why no floating point?** IEEE 754 cannot represent 0.1 exactly — currency arithmetic with floats introduces rounding drift. At 1M transactions, a 0.001% error = significant financial exposure
- **What is debit = credit invariant?** The double-entry principle: every financial event has equal and opposite effects. Sum of all debits across all accounts = sum of all credits. This is verifiable at any time
- **How do you prevent duplicate postings?** Idempotency key with UNIQUE constraint — second attempt hits the constraint, returns existing posting instead of creating duplicate
- **How do you handle partial failure in multi-leg posting?** Wrap all INSERT statements in a single DB transaction. Either all legs commit or none do
- **What happens if the snapshot is stale?** Two strategies: (1) tolerate eventual consistency for reads, (2) use optimistic locking — check snapshot version before serving balance to a payment authorizer
- **How does this differ from a regular transaction table?** A transaction table is a business record. A ledger journal is the accounting record — every financial event must appear here, even internal fund movements that have no user-visible transaction
